## Step 1 - Apply the data delta

The delta is wrapped in a single `BEGIN; ... COMMIT;` transaction.
Failure mid-stream rolls back the whole batch (no partial state).
Re-applying the same delta is a no-op (`ON CONFLICT DO UPDATE` for
systems / nodes, `ON CONFLICT DO NOTHING` for equivalences).

**Recommended (script, with MD5 check + prompt):**

```bash
PROD_DATABASE_URL="postgres://..." \
  bash scripts/apply_delta_dump.sh \
    data/wot_delta_2026-05-05_to_2026-05-07.sql \
    --md5 3a7c22e66ecbe40e046c17ca511ed5ac
```

**Unattended (CI pipeline, no prompt):**

```bash
PROD_DATABASE_URL="postgres://..." \
  bash scripts/apply_delta_dump.sh \
    data/wot_delta_2026-05-05_to_2026-05-07.sql \
    --md5 3a7c22e66ecbe40e046c17ca511ed5ac \
    --yes
```

The script:
- Verifies the file exists and matches the expected MD5 (when `--md5`
  is supplied).
- Echoes the dump header (count of systems / nodes / equivalences).
- Echoes the connection target (host + DB name, no credentials).
- Pauses for confirmation (skip with `--yes`).
- Runs the SQL through `psql -v ON_ERROR_STOP=1`.
- Runs a post-apply verification query parsed from the dump header.

**Manual fallback:**

```bash
psql -v ON_ERROR_STOP=1 "$PROD_DATABASE_URL" \
  -f data/wot_delta_2026-05-05_to_2026-05-07.sql
```

**Compressed variant** (.gz, 4.1 MiB) if bandwidth is constrained:

```bash
gunzip -c data/wot_delta_2026-05-05_to_2026-05-07.sql.gz \
  | psql -v ON_ERROR_STOP=1 "$PROD_DATABASE_URL"
```

## Rollback

**Step 1 (data delta):** `INSERT ... ON CONFLICT DO UPDATE` cannot be
naively undone (we don't know what the prod row was before the
upsert). Two options:

1. *Restore from the pre-deploy snapshot* (cleanest, requires the
   pre-flight backup).
2. *Targeted DELETE* if and only if the 6 systems were not in prod
   before this deploy:

   ```sql
   -- Equivalences first (FK-safe)
   DELETE FROM equivalence
     WHERE source_system = 'apqc_pcf'
        OR target_system = 'apqc_pcf';

   DELETE FROM classification_node
     WHERE system_id IN ('geonames_features','schema_org','gs1_gpc',
                         'fibo','wordnet_nouns','apqc_pcf');

   DELETE FROM classification_system
     WHERE id IN ('geonames_features','schema_org','gs1_gpc',
                  'fibo','wordnet_nouns','apqc_pcf');
   ```

**Step 3 (code from `main`):** standard rollback - redeploy the
prior Cloud Run revision. The data delta and the code are
independent; rolling back code does not require touching the data.

## Safety properties (why this is safe)

- The delta is wrapped in one `BEGIN; ... COMMIT;`. Atomic.
- Every `INSERT` carries `ON CONFLICT ... DO UPDATE` (systems,
  nodes) or `DO NOTHING` (equivalences). Re-applying the same
  delta is a no-op.
- Auto-managed columns are excluded from the dump:
  `classification_node.id`, `classification_node.search_vector`,
  `classification_node.created_at`, `equivalence.id`,
  `equivalence.created_at`. Prod's `DEFAULT` clauses stay intact.
- Existing rows for unrelated systems are never read or touched.
- The delta dump was syntax-verified locally inside a rolled-back
  transaction (the systems INSERT compiles and executes cleanly).
- Code PRs were each landed with green CI and explicit test plans
  in their PR descriptions.
